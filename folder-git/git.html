Что такое Git?

Git — абсолютный лидер по популярности среди современных систем управления версиями. Это развитый проект с активной поддержкой и открытым исходным кодом. Система Git была изначально разработана в 2005 году Линусом Торвальдсом — создателем ядра операционной системы Linux. Git применяется для управления версиями в рамках колоссального количества проектов по разработке ПО, как коммерческих, так и с открытым исходным кодом. Система используется множеством профессиональных разработчиков программного обеспечения. Она превосходно работает под управлением различных операционных систем и может применяться со множеством интегрированных сред разработки (IDE).

Git — система управления версиями с распределенной архитектурой. В отличие от некогда популярных систем вроде CVS и Subversion (SVN), где полная история версий проекта доступна лишь в одном месте, в Git каждая рабочая копия кода сама по себе является репозиторием. Это позволяет всем разработчикам хранить историю изменений в полном объеме.

Разработка в Git ориентирована на обеспечение высокой производительности, безопасности и гибкости распределенной системы.

Репозитории

Создание и подключение к созданному удалённому репозиторию

1. Создание на сервере удалённого репозитория

2. git remote add origin ссылка на созданный репозиторий— подключение к удалённому репозиторию без клонирования

3. git push --set-upstream origin dev или -u origin dev— загрузить на сервер файлы (—set-upstream или -u — параметр на всякий случай, чтобы наш репозиторий на компе был связан с репозиторием на удалённом сервере)


git pull origin наименование ветки— команда для дальнейших загрузок с сервера в  без клонирования

Процесс подключения к удалённому репозиторию 1

git clone ссылка на репозиторий— копирование файлов на комп с сервера (подключение к удалённому репозиторию)

git status— показывает статус файлов в папке

git add . или имя-файла— индексирует изменения и даёт возможность откатить изменения

git commit -m "change gitignore" или "add index.html"— сохраняет (комметит) файл(ы) для репозитория (на компе)

git push --set-upstream origin dev или -u origin dev— загрузить на сервер файлы (—set-upstream или -u — параметр на всякий случай, чтобы наш репозиторий на компе был связан с репозиторием на удалённом сервере)

Процесс подключения к удалённому репозиторию 2

git repo— инициализация

git remote add origin ссылка на репозиторий— подключение к удалённому репозиторию без клонирования

git push --set-upstream origin dev или -u origin dev— загрузить на сервер файлы (—set-upstream или -u — параметр на всякий случай, чтобы наш репозиторий на компе был связан с репозиторием на удалённом сервере)


"repo"— это сокращение от «repoialize» (т.е. «инициализироать» или «включать», «запускать») Это значит, что когда Вы пишете «git repo», Git «включается» или «запускается» для данного репозитория (т.е. папки).
1 создание папки на компе
2 добавить хотя бы один файл
3 инициализировать в ней репозиторий
4 добавить в индекс и в комит
5 подключиться к удалённому репозиторию

Настраивание и подключение к удалённому репозиторию

git remote add origin ссылка на репозиторий— подключение к удалённому репозиторию без клонирования

git remote -v— путь к настроенному удалённому репозиторию

git remote remove origin— удаление действующего пути, после этого можно снова добавить, как в первой строке

Ветки

Создание ветки

git checkout -b название-ветки— создание ветки и переключение на созданную ветку

git branch— посмотреть какие есть локальные ветки в репозитории * будет указан текущая ветка git branch название-ветки — создание ветки

git branch -a — посмотреть какие есть локальные и удалённые ветки в репозитории * будет указан текущая ветка git branch название-ветки — создание ветки

git checkout название-ветки — переключение между ветками

git push —set-upstream origin dev или -u origin dev— загрузка новой ветки на удалённый репозиторий со всеми изменениями (commits)

git log — посмотреть изменения в ветках

Слияние веток

git merge ветка — вливание в ветку, в которой в данный момент находится пользователь (в команде указывается ветка, которая будет вливаться)

Запрос на слияние

pull request — GitHub

merge request — GitLab

Модели ветвления

1 Git Flow

master — главная ветка, которую оберегают от любых вливаний без проверки

develop (dev) — ветка, в которую вносятся все изменения

release — вливаются изменения для вливания в master после проверки

hotfix (hotfixes) — ветка для внедрения быстрых изменений

2 TBD — Trunk Based Development

master — основная ветка, в которую вливаются все изменения после проверки

доп. ветка — в которой производятся все изменения. Ветка удаляется после вливания или отклонения вливания в ветку master

Отмена слияний

git reset —merge коммит — до которого сбрасывается слияние. Отменяет только закоммиченные изменения.

git merge —abort — отмена слияния веток из-за их конфликта

Переименование ветки

git branch —m новое имя ветки — переименование ветки

Удаление ветки

Необходимо перейти из ветки на удаление в другую ветку

git branch —d ветка — удаление ветки

git branch —D ветка — удаление ветки в случае ошибки, что ветка не влита в другую ветка

Изменения и commits

Просмотр незакоммиченных изменений и сравнение изменений

git diff наименование файла/commits — просмотр изменений, также можно посмотреть без наименования файла/commits

Данной командой можно просмотреть любые изменения, в том числе сравнить закоммиченные изменения

1 Для сравнения изменений между коммитами необходимо вначале ввести команду «git log» и выбрать первые четыре цифры двух коммитов.
2 Далее ввести команду «git diff» и следом прописать 4 цифры более раннего коммита и потом 4 цифры более позднего коммита

Для сравнения изменений двух файлов необходимо после команды «git diff» ввести наименования файлов через пробел

Откладывание изменений

git stash — отложить внесённых изменений

git stash pop — вернуться к отложенным изменениям, тем самым данные изменения сливаются с последним commit

git stash list — посмотреть список отложенных изменений

Примечание: если имеется несколько отложенных изменений, то при восстановлении командой «git stash pop» восстановится последнее отложение изменение


git stash drop номер — удаление отложенного изменения


Примечание: возможно отложить изменения в одной ветке, а восстановить в другой

Исправление и добавление в последний коммит

git commit —amend —m ’комментарий’ — исправление названия последнего коммита

git commit —amend —no-edit — добавление изменённых файлов в последний коммит

Отмена закоммиченных изменений

git checkout коммит файл — отмена коммита по данному файлу

git revert commit коммит — отмена указанного коммита. После ввода данной команды будет предложено отредактировать коммит. Обычно просто выходят из редактора командой «wq» и нажимают Enter

git revert no-commit коммит — отмена указанного коммита. Даёт возможность удалить несколько коммитов подряд и после удаления всех коммитов в конце можно один раз закоммитить

Исключение папки/файла из отслеживаемых:

Вначале добавить файл в gitignore

git rm —cached файл — исключить файл из отслеживаемых

Сброс/откат изменений

git reset —soft коммит — возвращает изменения в индекс. На файлы в индексе не воздействует

git reset —mixed коммит — отменяет коммиты вплоть до указанного и возвращает все изменения в рабочую директорию

git reset HEAD~1 — указывает сколько последних коммитов надо сбросить, при этом сброшенные коммиты будут неиндексированы

git reset —hard коммит — сбрасывает до указанного коммита и удаляет сброшенные коммиты безвозвратно

git reset —hard — отмена изменений (воздействует на файлы, которые отслеживаются Git-ом (безвозвратно)

Откат незакоммиченного файла

git restore файл — откат незакоммиченного файла

git restore —staged файл — удаление из индекса незакоммиченного файла (безвозвратно)

Удаление файлы, неотслеживаемого Git-ом

git clean -f — удаляет файлы, которые не отслеживаются Git-ом (безвозвратно)

Удаление папки/файла

1 удалить из локальной папки

2 прописать git rm -r название папки/

3 git add название папки

4 git commit -m «commiteted название папки»

5 git push —set-upstream origin dev

Разрешение конфликтов

Meld— редактор для разрешения конфликтов

git config —global merge.tool meld — установка meld в git

Пример: при слиянии двух конфликтующих веток откроется редактор meld и предложит варианты для разрешения
